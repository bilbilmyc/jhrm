# ADR-0003: 14 系统的三层 scope（手写核心 + 程序化填充 + 延后扩展）

游戏有 14 个修炼系统，按"内容写作量"和"自动化潜力"分三层：

| 层 | 系统 | 数量 | v0.1 状态 |
|---|---|---|---|
| 手写核心 | 灵根 / 功法 / 道心 / 天劫 / 因果 / 寿元 | 6 | 在 v0.1 |
| 程序化填充 | 装备 / 丹药 / 灵兽 | 3 | 在 v0.1 |
| 延后扩展 | 道侣 / 声望 / 弟子 / 世界事件 / 转世/夺舍/轮回 | 5 | v0.2+ |

**程序化层**用模板 + 元素 + 品级规则生成实例（例：`剑` 模板 + `火` 元素 + `下品` 品级 → `火灵剑·下品`）。**手写层**每个系统有专属内容（事件、选项、规则）。**延后层**v0.1 留空或仅放占位 UI。

## 为什么

完整 修仙 游戏的"系统数"是 10+，如果全部手写内容实体，一个人 6-12 个月都写不完。程序化生成处理"量大但模式化"的系统（装备/丹药/灵兽都是"模板 × 元素 × 品级"），手写处理"有灵魂"的系统（剧情、选择、性格）。两层结合，9 个 v0.1 系统对应 50-80 个内容实体，单人可写。

## 后果

- v0.1 实际交付 9 个系统（6 手写 + 3 程序化），其余 5 个是 stub 或完全缺失
- **程序化系统的"规则"本身是内容**：模板清单、元素相互作用、品级曲线都要设计 + 调优
- v0.2+ 加延后系统是"加一个系统"的工作，不是"写 200 个条目"的工作
- 程序化系统的"种子"必须进存档（否则重启游戏装备/丹药/灵兽会变）
