# ADR-0004: MVP 故意做小（1 位面 / 1 境界 / 3 系统 / 1 boss）

MVP = 跑通核心游戏循环所需的最小切片：

- **1 位面**：凡界（~10 节点）
- **1 境界**：炼气期
- **3 系统**：灵根 / 功法 / 道心
- **1 Boss IF 段**
- **金手指能调出**（隐藏手势/密语工作）
- **1 存档槽**

预计工作量 **1-2 个月**。

## 为什么

完整 v0.1（4 位面 / 6 境界 / 9 系统 / 6 boss）= 3-4 个月。先出 1-2 个月的 MVP，开发者能玩、能找到"哪里好玩 / 哪里不 work"，再决定后续扩展方向。金手指让"小游戏也能完整测试"——不必等完整 v0.1 才有得玩。

## 考虑过的方案

| 档 | 范围 | 工时 |
|---|---|---|
| 完整 v0.1 | 4 位面 / 6 境界 / 9 系统 / 6 boss | 3-4 月 |
| 半 v0.1 | 2 位面 / 2 境界 / 5-6 系统 | 2-3 月 |
| **MVP（推荐）** | **1 位面 / 1 境界 / 3 系统 / 1 boss** | **1-2 月** |

## 后果

- MVP 是"小而完整"——可玩的修真闭环，不是技术 demo
- MVP 后扩展是**渐进**的：先加 灵界 + 筑基 + 寿元/因果；再装备/丹药/灵兽程序化；最后 道侣/声望/弟子/世界事件。每个 1-2 月一步
- MVP 的**数据模型 / 存档格式 / IF 段结构必须从一开始支持全 14 个系统**，避免以后重写
- MVP 选中的 3 个系统（灵根 / 功法 / 道心）覆盖了"角色 / 动词 / 选择"三个 IF 主轴，是修真感最浓的三个
